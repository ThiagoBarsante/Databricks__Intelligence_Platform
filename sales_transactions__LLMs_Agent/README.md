# Sales Transactions — Multi-Agent / Multi-LLM SDP benchmark

This repo is a controlled experiment: request  **the same task, the same single 
instruction file, and the creation of the same Spec-Driven Development (SDD) process** were given
independently to several **coding agent / frontier LLM** combinations. Each one
had to provide the Specs, build and deploy a Bronze → Silver → Gold medallion **Lakeflow Spark
Declarative Pipeline (SDP)** on Databricks from the
[sales transactions dataset](sales_transcation_data_raw), generating the SDD
artifacts *together with* the implementation — so both the pipelines and the
agents' working styles can be compared side by side (the project structure was also defined by the Agent itself).

(`requirements.md`, `design.md`, `tasks.md`) — no specs were written by hand —
along with the pipeline SQL the model produced and a README with the deployed
results requested after the deployment - build a README.md file at project root level with the result of this implementation.

## Source data

data-quality issues that any well-built pipeline is expected to detect and handle.

## Key Findings — Business Summary

Every run started from the same single instruction file (e.g. `Instrunctions.txt`)
describing the dataset and the SDD task, followed by one deploy prompt:
*"read the instructions and deploy the SDP solution at Databricks."* From that
single file, each agent independently read the data, inferred the business rules,
designed the Bronze/Silver/Gold pipeline, and deployed it end to end — with no
further human guidance during the build.

### 1. The biggest gap was in the Silver layer (data-quality enforcement)

| Model | Silver output (from 100,000 raw rows) | Identify and apply data-quality rules? |
|---|---|---|
| GPT-5.5 | 5,896 | Yes |
| Claude Opus 4.8 | 11,227 | Yes |
| Claude Fable 5 | 11,227 | Yes |
| Claude Sonnet 4.6 | 100,000 (passthrough) | No |
| GPT-5.3 (Codex) | 100,000 (passthrough) | No |

GPT-5.5, Opus 4.8, and Fable 5 analyse the dataset and provide a similar analysis provided at the data-quality notes documented in the
[raw dataset README](sales_transcation_data_raw/README.md) — invalid ages,
negative prices/quantities, nulls, inconsistent gender labels, impossible
ship dates — and built validation and cleansing rules into the Silver layer,
the way a production data platform should.

Sonnet 4.6 and GPT-5.3 (Codex) passed all 100,000 raw rows through unfiltered
and pushed deduplication down to Gold, effectively skipping the cleansing step.

**Action item:** future instructions must *explicitly* require the agent to
inspect and remediate the documented data-quality issues — it cannot be assumed
that a model will proactively look for this on its own, especially if the plan is to use lower-cost implementation models such as Codex 5.3 or Sonnet 4.6.

### 2. Two different deployment styles emerged

- **Anthropic models (via Claude Code or GitHub copilot):** built everything directly in the live
  Databricks workspace — catalogs, schemas, pipelines, and (in one case) a
  dashboard — created step by step through the agent's own tooling.
- **OpenAI models (Open AI Codex  or GitHub copilot):** packaged the solution as a
  **Databricks Asset Bundle (DAB)**, a version-controlled project validated and
  deployed with `databricks bundle deploy` or via `python databricks sdk` if requested.

Both are valid: the DAB approach is closer to a CI/CD-ready deliverable, while
the direct-build approach is faster to iterate on interactively.

### 3. SDD task tracking: only the OpenAI models kept the specs in sync

The instruction file was **identical for every model**, and it explicitly required
Spec-Driven Development *alongside* the implementation: generate
`requirements.md` / `design.md` / `tasks.md` and follow those steps while building
the pipeline — meaning the SDD artifacts were part of the deliverable, not just a
planning aid.

- **OpenAI models (GPT-5.3 and GPT-5.5, via Open AI Codex):** generated the SDD docs
  and **updated them as they executed** — each task was marked as completed in
  `tasks.md` during the run, so the specs finished as a faithful record of what
  was actually built and deployed.
- **Anthropic models (Sonnet 4.6, Opus 4.8, Fable 5, via Claude Code):** generated
  the same SDD docs, built and deployed the pipeline — but **did not update the
  specs to reflect completion**. The `tasks.md` checkboxes were left untouched, and
  a **manual follow-up prompt** was required to ask the agent to mark the completed
  work in the SDD artifacts.

Since the instruction was the same for all runs, this is a genuine behavioral
difference between the models, not an artifact of prompting.

**Action item:** until this behavior converges, instructions should explicitly
state that `tasks.md` must be updated as each step completes, with a final pass
after deployment and validation — and runs with Anthropic models should budget a
manual "update the SDD specs" prompt at the end.

### 4. Bottom line

Every model produced a working pipeline, and every model also had gaps — some
skipped data-quality rules, others still need packaging for production. None of
the outputs should be deployed as-is: a data engineer should review the results,
combine the strongest parts of each, and harden the final solution.

What this exercise demonstrates is the real value: **a single instruction file
plus a frontier-model coding agent can scaffold an entire data platform solution
from scratch** — schema design, pipeline code, data-quality logic, deployment,
and validation — in minutes. The design choices and insights these agents surface
are a strong accelerator for Data Engineering and Analytics teams, even though
expert review remains essential before anything goes to production.

### 7. Final Consideration

For this experiment, **Claude Opus 4.8** produced the strongest overall outcome, particularly in terms of **data-quality refinement**, **documentation quality**, **price** compared to Fable 5, and the final generated deliverables. One notable observation was that Opus 4.8 was able to improve data quality while preserving a larger portion of the original information, avoiding excessive filtering during the cleansing process.

As a result, it was selected as the foundation for further development. Additional scripts were subsequently created to **automate the data-pipeline audit process** and **ingest new incoming data without manual intervention**. The extended workflow was also generated by AI agents, with **Opus 4.8 acting as the Agent Manager** coordinating the implementation.

That said, it is important to recognize that this conclusion reflects only the results of a **small, controlled benchmark**. Unlike traditional deterministic software, Large Language Models (LLMs) are inherently probabilistic systems. Small variations in prompts, instructions, context, or evaluation criteria can lead to significantly different outcomes. A more detailed or better-structured prompt, for example, could potentially shift the ranking and position another model—such as **GPT 5.5**—as the top-performing solution for the same task.

Therefore, these findings should be interpreted as a point-in-time comparison rather than a definitive ranking of model capabilities. Independent validation, expert review, and task-specific benchmarking remain essential before adopting any AI-generated solution in a production environment.
